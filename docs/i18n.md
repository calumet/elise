# Internacionalización (i18n)

`@calumet/elise-i18n` provee internacionalización y formateo localizado para Elise. Funciona tanto **standalone** (tu app traduce su propio texto) como **integrado con `elise-ui` y hermanos** (las labels internas de los componentes Elise se traducen automáticamente).

## Características

- **Cero deps externas en runtime**. Construido sobre React Context + `Intl.*` nativos.
- **Hooks**: `useTranslation()`, `useLocale()`.
- **Interpolación**: `t("greeting", { name: "Ana" })` resuelve `"Hola {name}"` → `"Hola Ana"`.
- **Formatters**: `formatDate`, `formatDateRange`, `useDateRange`, `formatNumber`, `formatCurrency`, `formatPercent`.
- **Integración transparente** con `elise-tables`, `elise-alerts`, `elise-toasts` (y futuros `elise-ui` widgets) vía un Context compartido por `Symbol.for`.

## Instalación

```bash
pnpm add @calumet/elise-i18n
```

## Setup mínimo

```tsx
import { I18nProvider } from "@calumet/elise-i18n";

const messages = {
  "es-CO": {
    app: { hello: "Hola, {name}" },
  },
  "en-US": {
    app: { hello: "Hi, {name}" },
  },
};

function App() {
  return (
    <I18nProvider defaultLocale="es-CO" fallbackLocale="en-US" messages={messages}>
      <YourApp />
    </I18nProvider>
  );
}
```

## Usar traducciones

```tsx
import { useTranslation, useLocale } from "@calumet/elise-i18n";

function Greeting() {
  const { t, locale } = useTranslation("app");

  return (
    <>
      <p>{t("hello", { name: "Ana" })}</p>
      <p>Locale activo: {locale}</p>
    </>
  );
}

function LocaleToggle() {
  const { locale, setLocale } = useLocale();
  const next = locale === "es-CO" ? "en-US" : "es-CO";
  return <button onClick={() => setLocale(next)}>{locale}</button>;
}
```

## Fallback sin Provider

Si llamas `useTranslation()` sin `<I18nProvider>` montado, `t(key, { fallback })` retorna el fallback (o la key si no se da). Esto es útil para librerías que quieren consumir traducciones cuando están disponibles, sin obligar a sus consumidores a montar un provider.

## Integración con los paquetes de Elise

`elise-tables`, `elise-alerts` y `elise-toasts` tienen strings internos (etiquetas, aria-labels, placeholders) que se traducen **automáticamente** cuando hay `<I18nProvider>` montado. Cada paquete reserva un namespace:

| Namespace | Paquete        | Ejemplos de keys                              |
| --------- | -------------- | --------------------------------------------- |
| `tables`  | `elise-tables` | `noData`, `rowsPerPage`, `firstPage`, `clear` |
| `alerts`  | `elise-alerts` | `ok`, `confirm`, `cancel`                     |
| `toasts`  | `elise-toasts` | `close`                                       |

Para sobrescribir las labels en español:

```ts
const messages = {
  "es-CO": {
    tables: {
      noData: "No hay datos para mostrar.",
      rowsPerPage: "Filas por página:",
      firstPage: "Ir a la primera página",
      // ... resto de keys
    },
    alerts: { ok: "Aceptar", confirm: "Confirmar", cancel: "Cancelar" },
    toasts: { close: "Cerrar" },
  },
};
```

Si no defines una key, el componente usa su fallback hardcoded en inglés. Tampoco hay error si no instalas `@calumet/elise-i18n` — los componentes siguen funcionando con los fallbacks.

### ¿Cómo funciona la integración?

`elise-i18n` y los paquetes hermanos comparten un `React.Context` registrado en `globalThis[Symbol.for("@calumet/elise.i18n.context")]`. El primer paquete que se evalúa crea el Context; los demás resuelven la **misma instancia**. Cada componente lee el Context con un `useContext` y, si está vacío, retorna el fallback. Resultado: integración automática sin acoplamiento entre paquetes.

## Formateo localizado

### Fechas

```tsx
import { formatDate, formatDateRange, useDateRange } from "@calumet/elise-i18n/dates";

formatDate(new Date(), { locale: "es-CO", month: "long" });
// "27 de febrero de 2026"

formatDateRange({ from: new Date(2026, 0, 1), to: new Date(2026, 0, 31) });
```

### Números, moneda y porcentajes

```tsx
import { formatNumber, formatCurrency, formatPercent } from "@calumet/elise-i18n/numbers";

formatNumber(1234567.89, { locale: "es-CO" });
// "1.234.567,89"

formatCurrency(99.95, "USD", { locale: "en-US" });
// "$99.95"

formatPercent(0.42, { locale: "es-CO", maximumFractionDigits: 1 });
// "42 %"
```

Los formatters aceptan `locale` como opción. Si no se pasa, usan el default del navegador. Para que tomen el locale del Provider automáticamente, combínalos con `useLocale()`:

```tsx
const { locale } = useLocale();
return <span>{formatCurrency(99.95, "USD", { locale })}</span>;
```

## API de referencia

```tsx
type I18nProviderProps = {
  children: React.ReactNode;
  locale?: Locale; // controlled
  defaultLocale?: Locale; // uncontrolled initial
  fallbackLocale?: Locale; // usado cuando la key no existe en el locale activo
  messages: Messages;
  onLocaleChange?: (locale: Locale) => void;
};

type TranslateOptions = Record<string, string | number> & { fallback?: string };

function useTranslation(namespace?: string): {
  t: (key: string, options?: TranslateOptions) => string;
  locale: Locale;
};

function useLocale(): {
  locale: Locale;
  setLocale: (locale: Locale) => void;
  locales: Locale[];
};
```

## Limitaciones de v1

- Sin tipado estricto de keys (toda key es `string`).
- Sin plurales/ICU avanzado (usar interpolación simple `{var}`).
- Sin lazy loading de namespaces (los messages se pasan completos al Provider).
- SSR: probado solo en CSR (Vite). Next debería funcionar pero no está testeado.
